# Accessibility Notes — WCAG 2.2 AA target

The journey is designed keyboard-first; the mouse is optional everywhere.

## Implemented

| Requirement (WCAG) | Implementation |
|---|---|
| Keyboard operation (2.1.1), no traps (2.1.2) | Every action is a native `button`, `a`, `input`, `select`, or `details/summary`; the delete confirmation is inline (no focus-trapping modal); PDF evidence has a text equivalent so the canvas needs no interaction |
| Visible focus (2.4.7), focus appearance (2.4.13) | Global `:focus-visible` 3px high-contrast outline with 2px offset |
| Focus management on navigation (2.4.3) | Route changes move focus to `main` (tabindex="-1"); skip link is the first tabbable element |
| Status messages (4.1.3) | A single polite `aria-live` region announces extraction results, confirmations, recomputed income, export, deletion, and errors |
| Labels & instructions (3.3.2), error identification (3.3.1) | Every control has a programmatic label; errors render in `role="alert"` banners near the control; upload rules described via `aria-describedby` |
| No color-only status (1.4.1) | Every status chip pairs an icon/prefix with text: "✓ confirmed", "✋ abstained — needs your entry", "⚠ NEEDS_REVIEW", "? Unknown" |
| Contrast (1.4.3) | Body 14.9:1, muted text 8.6:1, primary buttons 6.3:1, chips ≥ 6:1 — all above the 4.5:1 AA line |
| Target size (2.5.8) | Buttons min-height 40px; nav pills and row buttons ≥ 24px |
| Headings & landmarks (1.3.1, 2.4.6) | One `h1` per page, ordered `h2/h3`, `header/nav/main/footer` landmarks, labeled `nav`, `caption` on every table, `th scope` throughout |
| Non-text content (1.1.1) | The rendered PDF canvas is `role="img"` with a descriptive label; every highlighted box is also listed as text ("field, page, box coordinates") |
| Reduced motion (2.3.3 AAA courtesy) | `prefers-reduced-motion` disables all animation/transitions |
| Language & titles (3.1.1, 2.4.2) | `lang="en"`, descriptive `<title>` |
| Consistent navigation (3.2.3) | The stepper is identical on every page; the current step is `aria-current="page"` |

## How to verify

1. **Keyboard-only pass:** unplug the mouse; Tab from the address bar through Landing → consent → upload → confirm each field → Understand → ask a question → Prepare → export → delete. Every control must be reachable, operable with Enter/Space, and show a visible focus ring.
2. **Screen reader spot-check (NVDA/Narrator):** upload a document — the extraction result is announced; correct a field — the recomputed income is announced; delete — the confirmation is announced.
3. **axe DevTools scan** on each of the five routes: target 0 critical/serious issues.
4. **Zoom to 200%** — layout is single-column responsive with no horizontal scroll except inside data tables.

## Known gaps (honest list)

- The PDF canvas itself is not zoom/pan operable by keyboard; equivalence is provided as structured text rather than canvas interaction.
- Automated axe runs were not scripted into CI within hackathon time; the manual protocol above is documented in the demo script.
